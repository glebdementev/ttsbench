# Композитная оценка: взвешенное геометрическое среднее

## Формула

```
score = score_clicks^0.27 * score_noise^0.13 * score_robotic^0.20 * score_bandwidth^0.20 * score_pitch^0.20
```

Каждая суб-оценка — шкала 0-100. Показатели степени — это веса детекторов, в сумме дающие 1.0,
поэтому композитная оценка остаётся на той же шкале 0-100.

Эквивалентная запись в логарифмическом пространстве (так считает код):

```
score = exp( 0.27*ln(clicks) + 0.13*ln(noise) + 0.20*ln(robotic) + 0.20*ln(bandwidth) + 0.20*ln(pitch) )
```

## Почему геометрическое среднее, а не арифметическое?

Арифметическое среднее (простое взвешенное) — **компенсаторное**: оценка 22 за полосу пропускания
может быть компенсирована оценкой 100 за клики, и итоговый балл скроет очевидную проблему.
В качестве аудио одно плохое измерение отчётливо слышно, независимо от того, насколько хороши
остальные параметры.

Геометрическое среднее — **некомпенсаторное**: снижение одного фактора непропорционально
тянет вниз всё произведение. Это соответствует человеческому восприятию — аудио с идеальной
просодией, но сильным шумом всё равно звучит плохо.

### Конкретный пример

| Детектор   | Оценка |
|------------|--------|
| clicks     | 100    |
| noise      | 55     |
| robotic    | 95     |
| bandwidth  | 80     |
| pitch      | 90     |

- Арифметическое среднее (взвешенное): `100*0.27 + 55*0.13 + 95*0.20 + 80*0.20 + 90*0.20 = 87.4`
- Геометрическое среднее (взвешенное): `100^0.27 * 55^0.13 * 95^0.20 * 80^0.20 * 90^0.20 = 86.0`

Разница здесь невелика. Но при более экстремальном выбросе (bandwidth = 22):

- Арифметическое: **66.2**
- Геометрическое: **58.7**

Геометрическое среднее более агрессивно выявляет проблему — именно это нам и нужно.

### Интуиция

Представьте умножение вместо сложения. Если один множитель близок к нулю —
произведение схлопывается, нельзя «добавить баллов» из других измерений.
Тот же принцип используется в Индексе человеческого развития (HDI) и большинстве
многомерных индексов качества: каждое измерение — необходимое условие, а не бонус.

## Краевой случай: нулевые оценки

Оценка ровно 0 обнулила бы всё произведение. Мы ограничиваем минимум значением 0.1
перед взятием логарифма, поэтому нулевая оценка одного детектора даёт композит около 0,
но не создаёт -infinity.
